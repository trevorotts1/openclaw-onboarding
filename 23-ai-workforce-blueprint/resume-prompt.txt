═══════════════════════════════════════════════════════════════════
RULE 8 - NEVER STOP (binding for every resume fire)
═══════════════════════════════════════════════════════════════════
- NEVER call sessions_yield, and NEVER end your turn declaring the build "done" unless verify-zhc-standard.sh exits 0 (libraries substantive on disk + closeout confirmed). "No sub-agents running" is NOT done.
- On a timeout / 429 / model failure: do NOT stop. Back off ~2 hours, retry; if it still fails, +2h and retry again - keep going. The recurring cron is your retry engine; it is idempotent and will re-fire.
- The ONLY legitimate stop is a REAL terminal state: every department done AND roleLibraryStatus=done AND sopLibraryStatus=done AND closeout confirmed (messagesDelivered non-empty). The shell guard removes the cron only then.
- After K failed attempts, escalate to Rescue Rangers (the shell guard does this for you by POSTing to the n8n webhook $RESCUE_RANGERS_WEBHOOK_URL - NOT a bot-to-bot Telegram group post, which bots cannot read) and KEEP RETRYING in slow-backoff. The client must NEVER discover the job wasn't finished.

STEP 0 - DECIDE WHICH FLOW YOU'RE IN
═══════════════════════════════════════════════════════════════════

THIS SELF-PING IS AN INTERNAL BUILD TRIGGER, NOT A NEW USER REQUEST AND NOT A CUE TO RE-READ OWNER CHAT.
You read STATE and you BUILD. Do NOT re-open, re-read, or re-analyze the owner's conversation
history / chat logs to "figure out what you know"; that is the exact drift this prompt prevents.
The build content comes from the saved interview answers + the role/SOP libraries on disk, never
from mining the owner's chat. (See INSTRUCTIONS.md "Context Ingestion + Pull-Forward Rule": context
ingestion was a bounded one-shot pre-pass; the resume layer never reopens it.)

Read `/data/.openclaw/workspace/.workforce-build-state.json` (Mac: `~/.openclaw/workspace/.workforce-build-state.json`).

Decision tree:

A. If `buildCompletedAt` is null AND any department has `status` in {pending, failed, stale-building} → CONTINUE TO BUILD FLOW (below).

A2. If ALL departments are `done` BUT `roleLibraryStatus` is not `done` OR `sopLibraryStatus` is not `done` (and `buildCompletedAt` is still null) → SKIP to LIBRARY FLOW. The role library / SOP library were not fully pulled in. This gate runs BEFORE closeout - a workforce is NOT complete until both libraries are populated.

B. If `buildCompletedAt` is set AND `closeoutStatus` is not in {done, sent} → SKIP to CLOSEOUT FLOW.

C. If both are clean (buildCompletedAt set AND closeoutStatus in {done, sent}) → REMOVE THIS CRON, then exit silently.

   To remove the cron yourself (in case the shell guard at STEP -1 didn't
   already do it - e.g. older onboarding bundle without the script):

   ```bash
   CRON_UUID=$(openclaw cron list 2>/dev/null \
     | awk '/workforce-build-resume/ { for (i=1;i<=NF;i++) if ($i ~ /^[0-9a-fA-F-]{8,}$/) { print $i; exit } }')
   [ -n "$CRON_UUID" ] && openclaw cron rm "$CRON_UUID"
   ```

   This is MANDATORY when state is fully terminal. Failing to remove the
   cron burns DeepSeek-V4-Pro tokens every 15 min for no reason - that is
   the exact bug v10.14.36 was shipped to prevent. Do not skip this step.

BUILD FLOW (Skill 23 - per the Post-Interview Handoff Protocol)
═══════════════════════════════════════════════════════════════════

1. Identify departments where `status == "pending"` or `status == "failed"` or (`status == "building"` AND `lastAttemptAt` older than 15 minutes - that means the prior build session died).

2. For each such department, BEFORE starting work:
   a. Flip its `status` to `"building"` and set `lastAttemptAt` to now (UTC ISO-8601).
   b. Write the state file back to disk atomically (tmp + rename).

3. Generate the department per the standard Skill 23 build (read `workforce-interview-answers.md` or the equivalent v2.0/v10.14.14 dossier, DO NOT re-interview, and DO NOT mine the owner's chat history as a content source; the saved answers + the role/SOP libraries are the only sources).

4. After EACH department finishes (or fails), update its entry:
   - `status` → `"done"` (success) or `"failed"` (with `failureReason`)
   - `completedAt` → now
   - `filesPresent` → list of materialized files
   - Persist state file immediately

5. After ALL departments are `done`, run the LIBRARY GATE before anything else (v10.15.8). Do NOT write `buildCompletedAt` until it passes:
   a. Run the gate:
      ```bash
      bash /data/.openclaw/skills/23-ai-workforce-blueprint/scripts/verify-library-gate.sh \
        || bash "$HOME/.openclaw/skills/23-ai-workforce-blueprint/scripts/verify-library-gate.sh"
      ```
   b. If it exits non-zero (role library and/or SOP library not populated) → go to LIBRARY FLOW, re-pull, and re-run until it exits 0.
   c. ONLY when the gate exits 0 (`roleLibraryStatus == done` AND `sopLibraryStatus == done`): write `buildCompletedAt` to state file (UTC ISO-8601).
   d. Set `closeoutStatus = "pending"`. This is the v10.14.17 contract - Skill 37 picks it up from here.
   e. Either invoke `~/.openclaw/skills/37-zhc-closeout/scripts/run-closeout.sh` inline (preferred - lowest owner-latency), OR exit and trust the next cron fire to dispatch a [CLOSEOUT-RESUME].

6. Do NOT message the owner during the resume - the owner only hears from you at Skill 37 Step 6 (the celebration delivery).

7. Do NOT increment `resumeAttempts` - the resume script already did that before invoking you. You only write department state.

Spawn parallel sub-agents (per the "Generation Orchestration" stages 3-4 of INSTRUCTIONS.md) to move fast - up to 10 dept agents concurrently, up to 50 role agents concurrently.

If you encounter ANY error you cannot recover from after 3 retries, mark that department `"failed"` with a `failureReason` and continue with the others. Don't block the whole build on one stuck department.

═══════════════════════════════════════════════════════════════════
LIBRARY FLOW (Skill 23 - ROLE LIBRARY + SOP LIBRARY gate, added v10.15.8)
═══════════════════════════════════════════════════════════════════

Every department is built, but the role library and/or the SOP library were not fully populated. This is the last-night incident class (multiple clients): workforces scaffolded, but `how-to.md` files never filled from the role library and SOP placeholders never authored. A workforce is NOT complete until BOTH are done. Do NOT write `buildCompletedAt` and do NOT start closeout while this gate is dirty.

1. Measure the gap:
   ```bash
   bash /data/.openclaw/skills/23-ai-workforce-blueprint/scripts/verify-library-gate.sh \
     || bash "$HOME/.openclaw/skills/23-ai-workforce-blueprint/scripts/verify-library-gate.sh"
   ```
   It writes `roleLibraryStatus`, `sopLibraryStatus`, per-dept `roleLibraryFilled` / `sopLibraryFilled`, and `libraryFailureReason` into the state file, then exits: 0 = both done; 2 = role library not done; 3 = SOP library not done; 4 = both not done; 5 = qc could not run.

2. If the ROLE LIBRARY is not done (exit 2 or 4): re-run the role-library pull so every role's `how-to.md` is filled from `templates/role-library/`:
   ```bash
   python3 /data/.openclaw/skills/23-ai-workforce-blueprint/scripts/post-build-role-workspaces.py \
     || python3 "$HOME/.openclaw/skills/23-ai-workforce-blueprint/scripts/post-build-role-workspaces.py"
   ```

3. If the SOP LIBRARY is not done (exit 3 or 4): re-run SOP population so every role's SOP stubs are authored:
   ```bash
   python3 /data/.openclaw/skills/23-ai-workforce-blueprint/scripts/populate-sops-from-manifest.py \
     || python3 "$HOME/.openclaw/skills/23-ai-workforce-blueprint/scripts/populate-sops-from-manifest.py"
   ```
   If SOP population returns "owner-input-required" (no model selected), resolve the model per INSTRUCTIONS.md, then re-run.

4. Re-run `verify-library-gate.sh` until it exits 0. ONLY THEN proceed to BUILD FLOW step 5c (write `buildCompletedAt` + `closeoutStatus=pending`).

5. Do NOT message the owner during the library flow - it is internal, like the build resume.

CLOSEOUT FLOW (Skill 37 - ZHC Closeout, added v10.14.17)
═══════════════════════════════════════════════════════════════════

The build is done. The owner has heard NOTHING from you since the interview. The closeout is what closes the loop with them - celebration message, infographics, video, Notion doc, Command Center URL.

1. Read `~/.openclaw/skills/37-zhc-closeout/INSTRUCTIONS.md` for the full state machine. Key state fields:
   - `closeoutStatus`: pending → generating → sent → done (failed = retries exhausted)
   - `commandCenterStatus`, `commandCenterUrl`
   - `infographic1Url`, `infographic2Url`, `celebrationVideoUrl`, `notionRootPageUrl`
   - `messagesDelivered`: array tracking which of the 6 Telegram messages have landed

2. Invoke `~/.openclaw/skills/37-zhc-closeout/scripts/run-closeout.sh`. The script is FULLY IDEMPOTENT:
   - Skips any step whose target URL field is already set.
   - Step 6 (Telegram delivery) skips any message already in `messagesDelivered`.
   - Safe to re-run on a partially-complete closeout.

3. If the script exits non-zero:
   - Read the latest line in `$OC_ROOT/workspace/.zhc-closeout.log` for the failure reason.
   - The script has already written `closeoutStatus = "failed"` + `closeoutFailureReason`.
   - Don't message the owner. The next cron fire will retry.

4. If the closeout has been retried 3+ times unsuccessfully (check `resumeAttempts` trend and `closeoutFailureReason` not changing), send a single escalation to the operator's Telegram chat (env.vars.OPERATOR_TELEGRAM_CHAT_ID, default 5252140759) AND to Rescue Rangers by POSTing to the n8n webhook $RESCUE_RANGERS_WEBHOOK_URL (NOT a bot-to-bot Telegram group post - bots cannot read other bots) with the state file path and the failure reason. Then KEEP RETRYING (Rule 8) - do NOT abandon the closeout. Back off ~2h between heavy retries; the next cron fire continues. The build is only "done" when verify-zhc-standard.sh exits 0.

   Resolve the chat ID at the shell:
   ```bash
   source "$(if [ -d /data/.openclaw ]; then echo /data/.openclaw; else echo $HOME/.openclaw; fi)/skills/shared-utils/operator-chat-id.sh"
   openclaw message send --channel telegram --target "$OPERATOR_CHAT_ID" --message "..."
   ```

5. Do NOT message the owner about ANY closeout failure. Only message the owner via Skill 37 Step 6 (the actual celebration delivery), which the script handles.
