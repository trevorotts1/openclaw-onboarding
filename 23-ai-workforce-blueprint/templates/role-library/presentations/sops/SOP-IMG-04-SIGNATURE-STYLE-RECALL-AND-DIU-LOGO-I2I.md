# SOP-IMG-04 - Signature Style Recall ("use Style 1") + DIU Logo-as-Image-to-Image Mechanics

**Cluster:** Image-Gen Mechanics + Design-Library (skill 45) Integration
**Status:** Reference SOP - makes saved-style recall actionable + specifies the DIU logo-as-image-to-image path. The SOP-DIU-607 alias system already exists; this SOP ties it into the Presentations recall flow and mirrors the live Presentations image-to-image logo path (slide-image-creator.md + AF-F7) onto DIU-generated imagery. Forward reference doctrine for the Style Analyst / Brand Steward / Generation Operator.
**Owner roles:** Style Analyst (alias resolution), Brand Steward (folds recalled style into STYLE BLOCK), Generation Operator / Slide Submitter (logo-I2I at generation)
**Master authority extended:** `SOP-DIU-607` (Named Styles / aliases / lookbook); `45-design-intelligence-library/library/_system/MASTER-SOP.md` §3.2 (`{LOGO_NOTE}` variable); `MODEL-SPECS.md` §4 (I2I + style-reference directive); SOP-IMG-01 (Kie call mechanics)
**Library-version pin:** SOP-DIU-607 v1.0; MASTER-SOP v1.0; MODEL-SPECS v1.2

---

## 0. WHY THIS SOP EXISTS

Two needs:

- **Signature style recall (concern 22):** a client can say "create this presentation for me" and "use signature style 1" / "my PowerPoint style X." The system must know their styles, resolve the named style to the EXACT saved card + version + frozen references, and apply it - with zero guessing. SOP-DIU-607 fully specifies the alias system; this SOP makes recall actionable from the Presentations side and ties it to the conversation branch (SOP-IMG-03) and the seed file.
- **GAP c - DIU logo-as-image-to-image mechanics unspecified.** The Presentations pipeline has a clear I2I-with-logo-URL pattern (CLIENT-WEBINAR-DECK-SOP §9.1; SOP-IMG-01 Mode B). The DIU's `{LOGO_NOTE}` variable (MASTER-SOP §3.2) is TEXT-ONLY - there is no equivalent "pass the logo URL as a reference" pattern for DIU-generated style imagery. So a client whose logo must land consistently on DIU-generated brand imagery has no defined I2I mechanism. This SOP specifies it, mirroring the Presentations pattern.

This is build doctrine. None of it is printed on a slide.

---

## 1. PURPOSE

Make "use Style 1" resolve deterministically to an exact saved style and apply it; and give the DIU a defined logo-as-I2I mechanic so a saved style's logo is composited (not redrawn) - closing the logo-mutation defect for DIU-generated imagery the same way SOP-IMG-01 closes it for Presentations slides.

## 2. SIGNATURE STYLE RECALL - the resolution path (actionable)

When intake sets `STYLE_SOURCE = saved_style` and `STYLE_ID = "<name>"` (from the SOP-IMG-03 branch), recall runs:

1. **Resolve the alias.** The Style Analyst reads `_local/NAMED-STYLES.md` (the file seeded in SOP-IMG-03 §4) and finds the alias whose `alias` field matches `STYLE_ID` (plain-English match: "Style 1", "Signature Style 1", "Bold Dark Executive"). The alias yields: `card_id`, `card_version` (pinned), `frozen_refs`, `brand_overrides`.
2. **Verify production status.** Look up `card_id` in INDEX.md. It MUST be `status: production`. A draft/tested/retired card cannot be recalled - block and route to CDO (SOP-DIU-607 §A.2).
3. **Hand the resolved style to the Brand Steward.** Via SOP-DIU-612 Crossing A, the resolved `card_id@version` + the Foundation Prompt Block + `brand_overrides` are returned. The Brand Steward folds the Foundation Prompt Block into `style_block.md` (exactly as in SOP-IMG-02 §3, but with NO new analysis - the card already exists).
4. **Pin the version for the run.** Record `style_card_id@version` in `brand_registry.json`. The pinned version governs the whole deck (a card v2.0 bump mid-run is blocked by SOP-DIU-607 §B until CDO confirms + regression renders).
5. **Generate with the pinned style** (Phase 2 prompts inherit the foundation; Phase 4 generates per SOP-IMG-01).

**The client-facing menu:** the client chooses by LOOKING, not by reading markdown. The Style Lookbook (`_local/LOOKBOOK.md`, regenerated by SOP-DIU-607 §C) shows each named style with a thumbnail and "best for" line. "Use Style 1" maps to the Lookbook entry the client saw. If the client says a name not in NAMED-STYLES.md, the agent shows the Lookbook and asks which one - it never guesses an alias.

**Hard rule:** recall NEVER reconstructs a style from memory or from old generation logs. If the alias is not in NAMED-STYLES.md, the style is not recallable - fall back to the SOP-IMG-03 match-reference or creative-develop branch. Guessing "Style 1 was probably the gold one" is an auto-fail.

## 3. DIU LOGO-AS-IMAGE-TO-IMAGE (closes GAP c)

The DIU's style cards carry an optional `{LOGO_NOTE}` text variable (MASTER-SOP §3.2) - words only, which means the model REDRAWS the logo (the mutation path). This section adds the I2I mechanic so the logo is COMPOSITED from a supplied asset, mirroring SOP-IMG-01 Mode B.

**The mechanic (for any DIU generation where a client logo must appear):**

1. **The logo is passed as a reference URL, not described in words.** When `{LOGO_NOTE}` indicates a logo must appear AND a `LOGO_URL` is on file, the Generation Operator routes to an I2I-capable endpoint (`gpt-image-2-image-to-image` first choice; Nano Banana 2 backup - both accept reference images per MODEL-SPECS §1) and passes `LOGO_URL` in the reference slot (`input_urls` for GPT-Image-2 I2I; `image_input` for Nano Banana 2).
2. **The logo reference is named + "place, do not redraw."** The prompt states: *"The [first] reference image is the company logo: place it [per the card's logo placement], do not redraw, recolor, or restyle it."* - identical anti-mutation instruction to SOP-IMG-01 Mode B rule 3.
3. **The logo reference is NOT a style reference.** The MODEL-SPECS §4 style-reference-only directive ("use only as style reference ... do not copy subjects/faces/text") is NEVER applied to the logo - that would erase the logo's shape. If the same call also carries a STYLE reference frame, the two references get OPPOSITE instructions, each named: logo = "place exactly"; style frame = "style reference only." (Same rule as SOP-IMG-01 Mode B rules 4-5.)
4. **`{LOGO_NOTE}` text remains for the case with no asset.** If no `LOGO_URL` exists, `{LOGO_NOTE}` stays text-only and the logo is described (and WILL be approximate - flag this to the client; recommend supplying a logo asset for consistency). The I2I path is preferred whenever an asset exists.
5. **Reference reachability + limits** are identical to SOP-IMG-01 Mode B rule 2 (public https, 200, <=30 MB, count within the endpoint's reference cap: 16 for GPT-Image-2 I2I, 14 for Nano Banana 2).

**This makes the DIU logo path consistent with the Presentations logo path:** in BOTH pipelines, a supplied logo is composited image-to-image from a locked asset URL with a "place, do not redraw" instruction - never text-to-image, never reinvented per slide. A saved signature style that includes the client's logo therefore renders the SAME mark every time.

## 4. ENFORCEMENT CHECKS (auto-fail conditions)

| # | Check (trigger) | PASS | AUTO-FAIL |
|---|---|---|---|
| 1 | **Alias resolved from file, not memory.** When `STYLE_SOURCE = saved_style`, the style came from a matching alias in `_local/NAMED-STYLES.md`. | Alias matched in file | Style reconstructed from memory / old logs / a guess |
| 2 | **Production status.** The recalled `card_id` is `status: production` in INDEX.md. | Production | Draft / tested / retired card recalled |
| 3 | **Version pinned.** `style_card_id@version` recorded in brand_registry.json and held for the whole run. | Pinned + held | Card version drifted mid-run without CDO confirm + regression render |
| 4 | **Unknown alias handled.** If `STYLE_ID` is not in NAMED-STYLES.md, the agent showed the Lookbook and asked - it did not guess. | Lookbook shown / re-asked | Guessed which saved style the client meant |
| 5 | **DIU logo composited, not redrawn.** When a DIU generation must include a client logo and a `LOGO_URL` exists, the logo is passed as a reference (I2I) with the "place, do not redraw" sentence - not described in `{LOGO_NOTE}` text alone. | Logo as I2I reference | Logo only in `{LOGO_NOTE}` text while an asset existed (mutation path) |
| 6 | **Opposite directives kept distinct.** The style-reference-only directive is applied to the style frame ONLY, never to the logo or a face. | Scoped correctly | Directive applied to the logo/face (erasing them) |
| 7 | **Recalled logo identity.** A recalled signature style renders the same logo mark every time (matches the frozen ref). | Identical mark | A different mark than the saved style's frozen logo |

## 5. ESCALATION / REPAIR PATH

| Condition | First action | If unresolved |
|---|---|---|
| `STYLE_ID` not found in NAMED-STYLES.md (check 4) | Style Analyst regenerates/serves the Lookbook (SOP-DIU-607 §C) and asks the client to pick by looking. If they have none, route to SOP-IMG-03 creative-develop. | CDO |
| Recalled card not in production (check 2) | Block recall; CDO decides promotion (SOP-DIU-607 §A.2). Do not recall a non-production card. | CDO |
| Client requests a v2.0 of their saved style mid-run | Block the advance; SOP-DIU-607 §B requires a side-by-side regression render vs frozen refs + CDO written confirmation before the alias moves. The current run finishes on the pinned version. | CDO |
| DIU logo rendered as a wrong/mutated mark (check 5/7) | Re-route to I2I with the `LOGO_URL` reference + "place, do not redraw"; if it still garbles after 2 attempts, composite the real logo natively in post (mirror SOP-IMG-01 §8 fallback). | CDO |
| No `LOGO_URL` exists but client wants the logo on DIU imagery | Brand Steward requests a high-res transparent PNG + a public https URL (host to GHL media library / Drive); until then, flag that the logo will be approximate. | Director / operator |

## 6. PASS vs FAIL EXAMPLES (drawn from the forensic reference deck defects + the recall vision)

**FAIL (logo mutation, DIU side):** A client's saved "Bold Executive" style includes their logo, described only in `{LOGO_NOTE}` text. Each DIU-generated brand image redrew the logo differently - the same class of defect as the forensic reference deck's deck-wide logo mutation, now on brand imagery. Fails check 5.

**PASS (recall + composited logo):** Client says "make my new pitch deck, use Style 1." `STYLE_SOURCE = saved_style`, `STYLE_ID = Style 1`. Style Analyst matches the `Style 1` alias in NAMED-STYLES.md -> `PPT-002-C@v1.0`, production. Brand Steward folds the Foundation Prompt Block into the STYLE BLOCK, pins the version. Generation passes the client's locked `LOGO_URL` as the first reference with "place, do not redraw" - the same logo mark renders on every slide. Passes checks 1-7.

**FAIL (guessed alias):** Client says "use my premium one." No alias named "premium" exists. The agent guessed it meant the gold style and built the whole deck on the wrong saved style. Fails checks 1 and 4 (should have shown the Lookbook and asked).

**PASS (unknown alias handled):** Same "use my premium one." Agent: "I don't have one saved by that name - here's your style menu [Lookbook with thumbnails]; which of these, or should I develop a new one?" Client picks "Signature Style 2." Resolves cleanly. Passes check 4.

**FAIL (version drift):** Mid-run, the saved style's card was bumped to v2.0 by another job; the back half of the deck silently rendered in the new version, so the deck was visually inconsistent. Fails check 3 (the run should have held the pinned version).
