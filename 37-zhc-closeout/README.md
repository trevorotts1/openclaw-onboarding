# Skill 37 -- ZHC Closeout

Delivers the complete Zero-Human Company final package to the client at the end of the workforce build.

## What the Client Receives at Closeout

The single `<name>.zerohumanworkforce.com` Cloudflare URL -- the client's live dashboard. Every agent knows this link because run-closeout.sh writes it into AGENTS.md + TOOLS.md at closeout time.

| 2. What Is a Zero-Human Workforce? | Explains the hierarchy: Owner, CEO Agent, Dept Heads, AI Employees. |

| 6. How to Use Your Command Center | Walkthrough: dashboard access, Kanban columns, creating tasks, talking to dept heads, 3-check rhythm. Includes the CC URL. |
| 7. Communication Hierarchy | Owner to CEO Agent to Dept Heads to AI Employees. |
| 8. Lean Six Sigma in Your ZHC | Lean (remove waste) + Six Sigma (reduce variation) + DMAIC cycle. |
| 9. Book-to-Persona System | How the workforce decides HOW to handle every task. |
| 10. Your First 7 Days | Day-by-day action plan for week 1. |

Booklet prose is defined in `templates/booklet-content.md` (editable; not locked in shell heredocs).

6-message celebration sequence (send-telegram-celebration.sh) + operator summary (send-operator-summary.sh). CC URL included in msg6.

## Notion Delivery Tiers (root-page-create-failed permanent fix)

The Notion booklet is built by `scripts/create-notion-closeout.sh` in two tiers. It
tries Tier 1 first and automatically falls back to Tier 2 when the client has no
Notion of their own. The tier used is recorded in build-state as `notionTier`.

### Why this exists
An **internal** Notion integration **cannot** create a workspace-root page. It can
only create pages **under a page explicitly shared with it**. A client who never
shared a page produces `notionFailureReason = "root-page-create-failed: API returned
no id"`. There is **no API bypass**. (The Notion public API also cannot toggle
"share to web" per page -- `public_url` is read-only; sharing is enabled in the UI.)

### Tier 1 -- client owns their Notion (preferred)
The **client's own** integration writes the 9/10-section ZHC page under a page the
client shared (true ownership). Established at onboarding by
`scripts/ensure-notion-parent-page.sh`, which pins an already-accessible page as
`NOTION_CLOSEOUT_PARENT_PAGE_ID`, or emits a one-time client instruction:

> Share any one Notion page with your 'ZHC' integration (Notion -> open a page ->
> ... -> Connections -> add ZHC), or set `NOTION_CLOSEOUT_PARENT_PAGE_ID` to a page
> id you've shared. It auto-completes on the next run -- nothing else to do.

If neither is satisfied and Tier 2 is not configured, the builder **fails clear and
resumable**: it stages the full booklet locally
(`~/.openclaw/workspace/.zhc-closeout-notion-staged.md`), records a resumable
blocker, and exits non-fatal so the rest of the closeout still completes. It
**never fabricates a Notion page.** The resume cron re-runs and publishes instantly
the moment a page is shared.

### Tier 2 -- agency fallback (client has no Notion)
When the client has no Notion / shared nothing **and** Tier 2 is configured, an
**agency** integration creates the client's ZHC as a **subpage** under one private
agency parent page, titled `"<Client> -- ZHC Closeout"`, and returns a **view-only**
public link to that subpage.

**Operator prerequisites (one-time -- placeholders only, never commit real values):**

1. **`ZHC_AGENCY_NOTION_TOKEN`** -- the **agency** Notion integration token, set in
   operator secrets (`~/.openclaw/secrets/.env`) or operator config. This is an
   **operator secret**: never a client token, never committed to the repo.
2. **`ZHC_AGENCY_NOTION_PARENT_PAGE_ID`** -- the agency parent page id. That parent
   page MUST be:
   - **shared with the agency integration** (so the integration can write subpages), and
   - **Published to web, VIEW-ONLY**, in the Notion UI (so each client subpage
     inherits a public view link -- the API cannot enable this; it is a one-time UI step), and
   - kept **PRIVATE to workspace members** (do **NOT** share it with people or
     teamspaces). Web-publish is independent of member-sharing, so the parent stays
     private to the agency while children are view-only public.

**Privacy / no-co-mingling guarantees:**
- Each client receives **only their own subpage's** view link. Published subpages
  do not link to the parent or siblings, so a client cannot navigate to the parent
  or reach another client's subpage.
- Per-subpage access is **view-only** (inherited from the web-published parent), so
  clients cannot edit or vandalize.
- If the parent has not been web-published yet, Tier 2 still builds the subpage and
  returns the standard URL, recording a `notion-tier2-parent-not-published` blocker
  with the one-time operator action -- it never fabricates a share state.

The universal agency parent page id is operator-provisioned and supplied via the
`ZHC_AGENCY_NOTION_PARENT_PAGE_ID` env var -- it is **not** hardcoded in the repo.

## Auto-Fire Determinism (Skill 23 -- Skill 37)

As of v12.6.0 the Skill 23 -> Skill 37 handoff is deterministic:

**PRIMARY path:** On the HOP-4 transition (when buildCompletedAt is written and closeoutStatus=pending), `resume-workforce-build.sh` directly launches `run-closeout.sh` in-process via `nohup bash ... &`. No Telegram delivery required.

**SECONDARY path (fallback):** The existing `openclaw message send` self-ping still fires as a secondary nudge. If the gateway has Telegram-offset corruption, the in-process exec has already fired.

Same logic applies in `resume-closeout-cron.sh` -- every cron fire does both.

`run-closeout.sh` is fully idempotent -- every step guards on whether its artifact already exists. A double-fire is safe.
