# Notion Client-Doc Standard (Skill 38)

The **client Notion setup doc** (a.k.a. the Client Reference Sheet / Quick-Start doc) is the single
document the client opens to wire OpenClaw into their GHL (Convert and Flow) account. It is generated by
`scripts/21-generate-client-reference-sheet.sh`, mirrored to the client's OWN Notion workspace, and
delivered to the client via Telegram. This file is the **single standard** for what that doc must
contain and the EXACT order, and the structure is **machine-enforced** by
`scripts/qc-reference-sheet.sh` (+ `--require-manual-fill`), `scripts/qc-notify-client-doc.sh`, and the
new `scripts/qc-notion-doc-standard.sh`.

> A live client was stranded by a setup doc that had NEITHER the bearer token NOR a copyable Raw Body —
> there was nothing to paste into GHL's Build-with-AI. This standard exists so that never recurs, and it
> is gated in CI.

---

## 0. EVERY CLIENT NOTION SETUP DOC MUST INCLUDE ALL OF THE FOLLOWING, IN THIS ORDER

> **THIS IS THE STANDARD. NON-NEGOTIABLE. NO EXCEPTIONS.** Every client setup doc the agent generates —
> for EVERY client, EVERY run — MUST contain ALL of the numbered items below, IN THIS ORDER. A doc missing
> ANY of these (or out of order) is INCOMPLETE and is machine-FAILED by
> `scripts/qc-reference-sheet.sh --require-manual-fill` + `scripts/qc-notion-doc-standard.sh` (CI +
> pre-handoff QC). Universal: zero personal/client data — the standard ships with generic placeholders only.

1. [ ] **Quick Start FIRST.** A section literally named **🚀 Quick Start** leads the doc — the copy-paste
       values come first, the deep explanation comes AFTER (Quick Start does NOT replace the explanation).
2. [ ] **Webhook URL — its own code block.** The hook URL `https://<PUBLIC_HOSTNAME>/hooks/<HOOK_NAME>`
       as a single copyable code block (no trailing slash; the `/hooks/` segment present).
3. [ ] **Authorization as TWO separate code blocks.** A GHL header has a **Key** box and a **Value** box,
       so Authorization is emitted as TWO copyable blocks — **block 1 = exactly `Authorization`** (paste
       into the Key/header-name box), **block 2 = exactly `Bearer <token>`** (value-only, paste into the
       Value box). **NEVER a combined `Authorization: Bearer <token>` block** — that is the bug (the client
       pastes the whole string into the Value box) and is machine-FAILED. The value block must be ONLY
       `Bearer <token>` — it must NOT repeat the word `Authorization`.
4. [ ] **Content-Type as two code blocks.** Same Key/Value split — **block 1 = `Content-Type`**,
       **block 2 = `application/json`** (value-only) — never a combined `Content-Type: application/json` block.
5. [ ] **The FLAT 23-key Raw Body — its own code block.** The canonical FLAT 23-key GHL Custom Webhook
       Raw Body as a ` ```json ` fenced (copyable) code block, per `references/ghl-raw-body-json-standard.md`
       (Standard 2): all 23 keys, FLAT, placeholder-free `messageTemplate`, `deliver:false`. Plus the
       **Method (POST)** as a copyable block.
6. [ ] **Build step.** (a) Create the **tags FIRST** (before the workflow, so filters reference REAL tags —
       a blank/non-existent tag in a "does not contain" filter is the known bug) with a Settings → Tags
       pointer; (b) **field-by-field manual Custom-Webhook fill** instructions (Method, URL, the two
       headers via Add item, the Raw Body); (c) a **"Build-with-AI builds the SHAPE only" warning** — it
       does NOT reliably populate the URL / Authorization / Content-Type / Raw Body, so the client MUST
       paste those by hand; (d) a **post-build VERIFY checklist** (trigger/tag-exists + custom webhook
       fields non-empty + Published-not-Draft, run AFTER Build-with-AI finishes).
7. [ ] **"Your Communication Playbooks" section.** With the **"Help me build a [purpose] playbook"** CTA
       (a copyable example, e.g. *"Help me build a missed-call follow-up playbook"*), the personal
       **trigger word** offer ("Alexa"/"Hey Siri" style), the **I Do / You Do** process (+ the ~15-30-min
       expectation), and the **brainstorm** walkthrough (the AI brainstorms using what it knows — NOT a
       50-question interrogation — then creates + stores + wires the matching Workflow AI prompt). Placed
       AFTER Quick Start and BEFORE the deep reference.
8. [ ] **VPS-vs-Mac install-considerations section.** The "⚙️ Things to consider when installing: VPS
       (Hostinger Docker) vs Mac mini" section covering BOTH targets (authoritative content:
       `references/vps-vs-mac-install-considerations.md`) — VPS: host `/docker/<project>/.env` +
       `docker compose up -d --force-recreate` + container `/data/.openclaw/secrets/.env` +
       `OPENCLAW_HOOKS_TOKEN`/hooks-token-rewrite-on-boot; Mac: provider keys in the `openclaw.json`
       top-level `env` block + `launchctl kickstart -k gui/$(id -u)/ai.openclaw.gateway`.
9. [ ] **How-it-works explanation LAST.** The complete Full Reference & Explanation section comes at the
       END — after every copy-paste value, the playbooks section, and the install considerations.
10. [ ] **Every copyable value in its own code block.** URL, both Authorization blocks, both Content-Type
        blocks, the Raw Body JSON, the Method — each is its own fenced code block with its own copy button.
        No copyable value lives in prose.
11. [ ] **MUST be delivered to the client via Telegram.** The doc LINK is SENT to the client via the
        OpenClaw gateway (`openclaw message send --channel telegram`) — never `curl api.telegram.org`.
        Generating the doc is NOT delivering it; delivery is a separate, gated step (Section 3).
12. [ ] **UNIVERSAL — no personal/client data.** The standard + the generated structure carry zero real
        personal/client identifiers; every real value is a generic placeholder
        (`<PUBLIC_HOSTNAME>`, `<HOOKS_TOKEN>`, `<CLIENT_BUSINESS_NAME>`, `<HOOK_NAME>`).

## 1. The order, at a glance

```
🚀 Quick Start
  → Webhook URL (own code block)
  → Method: POST (own code block)
  → Authorization KEY block ("Authorization") + Authorization VALUE block ("Bearer <token>")
  → Content-Type KEY block ("Content-Type") + Content-Type VALUE block ("application/json")
  → FLAT 23-key Raw Body (```json code block)
  → Build step: tags FIRST → manual Custom-Webhook fill → "Build-with-AI builds the SHAPE only" → post-build VERIFY
Your Communication Playbooks (CTA + trigger word + I-Do/You-Do + brainstorm)
⚙️ Things to consider when installing: VPS vs Mac
Full Reference & Explanation  (how-it-works LAST)
```

## 2. Where the doc lives (the client's OWN account — never co-mingle clients)

The human-readable doc is created in the CLIENT's own workspace, in the fallback order **Notion → Google
Docs → plain-text** (see `references/communications-playbook-standard.md` §4). Never place a client's doc
in a generic/operator or another client's workspace. For Notion, it is a subpage under the client's
designated parent page (`NOTION_PARENT_PAGE_ID` / `NOTION_PARENT_SEARCH`, integration shared with that
parent).

## 3. Delivery (MANDATORY, machine-enforced)

The install is NOT complete until the doc LINK has been SENT to the client via Telegram. This is gated by
`scripts/22-notify-client-doc.sh` (sends via the gateway, discovers the chat from the TRANSCRIPTS
`agents/*/sessions/*.jsonl` — the most-frequent non-operator id — LOUD-fails on no chat, records
`clientDocDelivered=true`). See `references/communications-playbook-standard.md` §8.

## 4. Enforcement

- **Structure + order:** `scripts/qc-reference-sheet.sh --require-manual-fill` drives the generator in an
  offline sandbox and FAILs unless the rendered doc carries — in order — the 🚀 Quick Start first + a full
  explanation after, the URL, the SEPARATE Authorization key + value blocks (value-only `Bearer <token>`,
  never combined), the Content-Type blocks, the ` ```json ` 23-key Raw Body, the create-tags-FIRST
  instruction, the manual Custom-Webhook fill, the post-build VERIFY, the "Your Communication Playbooks"
  section (CTA + trigger word + I-Do/You-Do + brainstorm), and the VPS-vs-Mac section.
- **This standard doc:** `scripts/qc-notion-doc-standard.sh` asserts THIS doc carries the ordered mandatory
  list (items 1-12) and composes `scripts/qc-reference-sheet.sh` so the build FAILS on any violation.
- **Delivery:** `scripts/qc-notify-client-doc.sh` enforces the mandatory Telegram delivery.
- All three run in `scripts/11-run-qc-checklist.sh` + CI `.github/workflows/qc-static.yml`.
