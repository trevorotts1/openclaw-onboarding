# Version Architecture

## How Versioning Works

There are TWO sides to version tracking:

### Publisher Side (GitHub)
- `version` file at repo root: the canonical published version
- `install.sh` ONBOARDING_VERSION variable: MUST match the root version file
- These are set by the repo maintainer when pushing updates
- Rule: NEVER update one without the other

### Client Side (Installed Machine)
- `~/.openclaw/skills/.onboarding-version`: the installed version
- Written by install.sh after a successful install or update
- This is what the update script checks against GitHub

### Update Check Flow
1. Update script downloads GitHub version file
2. Compares GitHub version vs ~/.openclaw/skills/.onboarding-version
3. If different: update available
4. If same: already up to date

### Per-Skill Versions
- Each skill has skill-version.txt in its folder
- These strings are INFORMATIONAL ONLY. They are not a sync signal and
  must never be used as one: canonical routinely edits a skill's contents
  without bumping its version, so a tree can carry a version identical to
  canonical while its contents differ.
- The root `update-skills.sh` therefore does NOT compare versions per skill.
  It replaces every skill tree unconditionally (`rm -rf` + `cp -r`) and then
  verifies the result by CONTENT.

### Drift Detection (content, not version)
- `~/.openclaw/skills/.onboarding-content-manifest.json` is the authoritative
  record of what is installed. It stores a sha256 digest per skill tree,
  computed by `scripts/skill-content-hash.sh` over `relpath:sha256(content)`
  for every non-excluded file.
- `update-skills.sh` writes it manifest-first / stamp-last: if the manifest
  cannot be built, committed, or re-read intact, the run exits non-zero and
  `.onboarding-version` is never written. A box is never stamped current
  without a matching manifest.
- `check-updates.sh` (A4) recomputes the digests and compares them against
  this manifest to report drift.
- Known scope limit: `skill-content-hash.sh` enumerates only numbered
  (`[0-9]*`) skill directories. `shared-utils/`, `universal-sops/` and
  `~/.openclaw/scripts` are NOT covered by the content gate.

### Retired: .skill-manifest.json
- `~/.openclaw/skills/.skill-manifest.json` was a version-string inventory
  written once by `install.sh` and regenerated by nothing. It froze at the
  version of the last full install and reported it forever.
- It had no code readers. It is no longer written, and `update-skills.sh`
  deletes it (and the orphaned `scripts/generate-manifest.sh`) from any box
  it runs on. Use `.onboarding-content-manifest.json` instead.
- For a human-readable version inventory, use `openclaw skills list` or read
  the per-skill `skill-version.txt` files directly.

### Rollback
- Before any update, the full skills folder is backed up
- Rollback restores the backup and resets .onboarding-version
- NOTE: there is no rollback script in this repo today. Rollback is a manual
  restore of the backup directory. An earlier version of this document
  claimed `.skill-manifest.json` was regenerated after rollback; no such code
  has ever existed.

### Version Format
- Repo level: vMAJOR.MINOR.PATCH (e.g., v8.1.0)
- Skill level: vMAJOR.MINOR.PATCH (e.g., v7.0.0)
- MAJOR: breaking changes, architecture rewrites
- MINOR: new features, non-breaking additions
- PATCH: bug fixes, doc updates
