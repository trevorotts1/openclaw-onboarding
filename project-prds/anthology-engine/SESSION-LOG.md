## THE ENTRY CONTRACT (binding on every agent, per WAVE-PLAN Global Rule 4, PRD Section 10, and the GOAL-COMMAND incremental persistence law)

THE MODEL: each agent appends its OWN completion the MOMENT its unit passes the Gate A build gate (8.5 on the fleet 10-category rubric) and is PUSHED. The SAME agent, in the SAME motion, ticks CHECKLIST.md, marks TODO.md DONE, and appends CHANGE-LOG.md — the four-file persistence quartet. Completed-and-pushed work is thereby NEVER lost to a session limit. An agent that pushed but did not log has NOT finished its unit.

Every completion entry carries, inside its Entry cell, these labeled fields in order:
1. WHAT: one or two sentences stating what was built or verified, naming the WAVE-PLAN unit (W0.1 through W7.3).
2. GATE A: the final rubric score and the round count; if any round failed, the failing score and the fix applied (example: "Gate A 8.9, round 2; round 1 scored 7.8 on category 6, fixed by adding the kill-and-resume drill"). Mechanical/unscored units (repo mechanics) instead list each mechanical gate run and its PASS/FAIL.
3. PUSH: repo (onboarding OR Command Center — name which; the Command Center repo rides its OWN serial train), branch, short commit hash, and pull request number if one exists. No push means the field reads "NOT PUSHED" and the entry states why.
4. EVIDENCE: absolute paths, prover/guard names run, fixture identifiers, probe results observed — labels and hashes only, never values.
5. PERSISTENCE: the CHECKLIST.md item(s) ticked, the TODO.md id(s) marked DONE, and confirmation the CHANGE-LOG.md row was appended.
6. REMAINING: what this agent carries forward, or NONE.

## ADDITIONAL REQUIRED ENTRY TYPES

- WAVE OPEN / WAVE CLOSE: the orchestrator appends one row when a wave dispatches (agent count, stagger plan) and one when the wave closes (every unit pushed or explicitly carried forward). Waves are never stacked; the newest WAVE OPEN without a matching WAVE CLOSE is the live wave.
- QC ROWS: each independent QC agent logs the units it scored and the scores given (the QC agent is never the author of the unit it scores; the two banks log separately).
- RE-DISPATCH: any failure or rate limit gets a row naming the failed agent, the cause, and the resumeFromRunId passed forward. The loop never stops; the log proves it.
- W0.2 INGESTION EVIDENCE (mandated by WAVE-PLAN W0.2): per ingested prompt — source JSON filename in <OPERATOR_HOME>/Downloads/anthology project/, extracted byte count, sha256 pin, zero-truncation proof, and the zero-remaining-[UNCHANGED] proof for the Book-to-HTML formatter (restored from the CSV HTML Book record), plus the 2,000-to-3,500 word-band normalization and the Make.com-to-canonical variable map. HASHES AND BYTE COUNTS ONLY — never the prompt bodies, never any JSON export content (the nine exports carry live keys and are NEVER committed or quoted).
- SECRET HYGIENE (W0.3): rotation rows by LABEL only (the cover-node Authorization header key; the customData payload key re-homed as an ENV credential by label). A secret VALUE in this file is an incident.
- CANARY DRILLS (W5.x, W7.1): observed results per drill on the operator box, including forced-failure outcomes, both-door gate proofs, and the review-to-done promotion observed ONLY via the independent QC scorer. Gate B content-gate scores logged here stay clearly labeled GATE B — the two gates are never conflated in a log row.
- MERGE TRAIN (Wave 6): the single merge-writer logs each of the ten train steps as its own row — including the annotated tag v19.0.0 (git tag -a) created BEFORE the merge, the update.sh registration plus corrected skill count, the content_sha restamp, and the fresh-clone verification with the JSON-export-absence scan.
- FREEZE AND HOLD (W7.2, W7.3): the freeze posture, the retirement criteria recorded but not executed, and the fleet-rollout HOLD (repo-only until the operator OK).
