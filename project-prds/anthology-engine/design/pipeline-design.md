## 1. Execution model: idempotent one-stage jobs, never a blocking run

LEGACY ran one giant blocking execution per participant with eleven Gmail sendAndWait gates capped at 45 days; the observed result is zombie waits dead since late 2025 and two fatal credit errors with no resume. The engine inverts this: an event arrives, intake_router.py (deterministic, no model call, under 2 seconds) loads the participant from the ledger by the composite key contact_id plus anthology_id, advances EXACTLY ONE stage, persists every artifact and transition through anthology_state.py (the sole ledger writer), and stops. Multi-month participant pauses are a normal ledger state at zero cost. Every job is idempotent: replaying any event against current state is an acknowledged no-op (dedup fingerprint: sha256 of contact_id, anthology_id, stage, payload body).

Event sources: (a) the inbound webhook route /hooks/anthology-intake (universal form and per-stage participant forms, per-client secret under label ANTHOLOGY_INTAKE_HOOK_SECRET); (b) producer gate actions on the Anthology board card (which shell anthology_state.py subcommands; the board holds no base credential); (c) participant gate actions on the token page reached by the emailed nudge deep link; (d) the Kie.ai callback relay (Skill 46); (e) the daily tick (smoke test, hold-queue aging, mirror reconcile, 7-day stuck re-nudge); (f) operator actions from the exceptions queue. Both doors of any gate (board card and nudge link/token page) hit the SAME endpoint and the same writer subcommand, with the door recorded on the approvals row.

## 2. Model routing (Layer 2 owns it; prompts never name models)

model_router.py is the ONE call site for every model turn. The tier map (config/model-map.template.json, resolved per box at install against the CLIENT's own providers and keys, labels resolved across all three env stores, live process env first):

| Tier | Work | Resolution chain, in order | Parameters |
|---|---|---|---|
| HEAVY-WRITER | S1 avatar bodies, S2 tone, S3 titles, S4 blurb and outline, S5 chapter, S6 rewrite, S9 introduction and matter | (1) GLM 5.2 on Ollama Cloud, provider id ollama-cloud, which REQUIRES baseUrl slotting, not apiKey slotting, maxTokens 65536; (2) OpenRouter GLM 5.2; (3) Gemini 3.5 Flash (operator-confirmed final fallback for THIS project; the Gemini key resolves under any of its three alias names); (4) chain exhausted: durable HOLD plus ONE deduped founder Telegram alert through the OpenClaw gateway, never bypassed | thinking high, temperature 0.3 |
| LIGHT | Extraction (aw-12 primary goal), classification, tool-shaped chores | Minimax V3 on Ollama Cloud, then its OpenRouter counterpart, then Gemini 3.5 Flash | defaults |
| JUDGE | Gate B semantic checks and the Tier 2 content rubric | Minimax V3 or Gemini 3.5 Flash, ALWAYS a different resolution than the tier that drafted the piece (QC independence law) | temperature 0 |
| LONGCTX (optional) | S9 whole-manuscript compile | DeepSeek V4 Pro or Kimi 2.6 (about 1M context), ONLY when the client configured a key; otherwise S9 chunks on HEAVY-WRITER | thinking high |
| IMAGE | S7 cover | Client's own Kie.ai, GPT-image-2, PORTRAIT 1024x1536, via Skills 07 and 46, against the Wave-0-VERIFIED text-to-image portrait endpoint (never the 16:9 presentation recipe) | per Skill 07 |

Router semantics: per-call pre-meter and post-meter through anthology-cost-ledger.py (per-deliverable token budgets shared across QC attempts, ceiling blocks the call); typed error classes (insufficient_credits, auth, rate_limit, timeout, refusal) with chain advance on retryable classes and durable hold on credits; DENY PATTERNS refuse any resolved model matching Anthropic identifier shapes at call time (claude-*, anthropic/*, vendor-prefixed variants), mirroring Skill 54's AF-AW-ANTHROPIC gate, with guard-no-anthropic-runtime.py enforcing the same law statically at merge; the model actually used is recorded honestly on the Artifact row. Sovereignty: the chain is the client's OWN accounts under their own labels; operator keys never land on client boxes; the engine never substitutes a client's expressed model choice. Prompts never name models; tiers are the only vocabulary above the router.

## 3. Web-search auto-detection (Avatar Questions 31 and 32)

search_detect.py runs once per box, caches to client state, re-detects on preflight: (1) enumerate the client's ENABLED OpenClaw web-search tools on the installed gateway (live-verified at W0.4; tool-registration schemas drift between gateway versions); (2) PREFER PERPLEXITY when present; (3) else the best available enabled search tool (Ollama Cloud web search and other enabled options ranked by recency and quota state); (4) else NO TOOL: the avatar deliverable is produced from Questions 1 to 30 with an honest limitation note in the operator run report, the participant run CONTINUES, and ONE deduped Telegram flag goes to the founder through the gateway. Exit codes: 0 tool detected and cached; 3 no tool (degrade-plus-flag path). GLM 5.2 cannot search natively, so the search pass is a Layer 2 TOOL STEP whose findings are injected into the pinned Question 31 and 32 prompt (pin aa-02) as context; the pinned prompt body is never edited beyond slot substitution; the engine never asks a model to fabricate links, and fabricated-link detection is Gate B Tier 1 check 10 (research claims must trace to the search pass output).

## 4. The per-gate Convert and Flow pipeline-stage update (legacy behavior KEPT, registry-driven)

At EVERY gate passage the engine fires the Convert and Flow pipeline-stage update the operator wants kept (PRD decision 3.9), never hardcoded: (1) provisioning AUTO-PROVISIONS the standard Anthology pipeline and stage set in the client's own Convert and Flow account through the CLIENT'S OWN private integration token (provision-anthology-client.sh; PRD 3.12 locked default; binding to a pre-existing pipeline only as an explicit override) and binds the stage map into the anthology registry (anthology_registry.py, caf_stage_map); (2) at each gate, anthology_state.py advance-stage triggers caf_delivery.py to push the control field contact.anthology_stage AND the per-anthology opportunity/pipeline stage sync resolved from caf_stage_map, keyed by contact_id, tenant-checked (payload location equals the registry binding), then read back and verified; (3) the legacy defect class (nine hardcoded stage-update nodes bound to one client's pipeline) is structurally impossible because no stage identifier exists outside the registry binding. This replaces LEGACY-MAIN's M8 node family.

## 5. Stage specifications S0 to S9 (buildable stages)

Each stage is one idempotent runner (stage_s1_avatar.py through stage_s9_assembly.py; S0 is intake_router.py): compose pinned prompts, call Layer 1 (Skill 54 via anthology-entry.sh) or the router, run provers, record artifacts, open the gate, stop. Stage-runner exit codes (SPEC 3.4): 0 stage complete and persisted; 2 prover failure (counts a QC attempt); 3 held (credits or callback); 5 unresolved prompt slot (AF-AE-SLOT-UNRESOLVED, fail-closed). Every transition goes through anthology_state.py; every billable call is metered; the per-gate pipeline-stage update of Section 4 fires at every gate; the participant's board card status mirrors stage_cursor via mc_board.py (fail-soft: board unreachability never blocks the pipeline; the ledger is the truth and the card reconciles on the daily tick).

Operating roles and spoken personas per stage (the PRD Section 13 normative table, stamped into the persona-matching config at wiring so the client's workforce self-invokes the right operator): S0 and S8, anthology-producer-orchestrator, no persona (deterministic code); S1, anthology-chapter-author speaking the Avatar Profiler (aa-01 to aa-03); S2, the Tone Analysts and Blender (tone core 04 to 08); S3, the Senior Title Strategist (aw-06); S4, the Blurb Copywriter (aw-07) and the Outline Architect (aw-08); S5, the Anthology Chapter Author (aw-09); S6, Dr. Margaret Thornfield, editorial revisionist (aw-10); S7, anthology-producer-orchestrator speaking the Senior Book-Cover Design Specialist (aw-11); all gates, nudges, and the readiness report, anthology-approvals-steward; S9, anthology-producer-orchestrator with the Anthology Editor voice (ae-01 to ae-04), subordinate to producer inputs; Gate B judge passes, anthology-approvals-steward as the independent Editorial Judge on the JUDGE tier.

### S0 INTAKE AND ROUTING (intake_router.py; no model call)
Input: the universal Convert and Flow form. Visible: name, email, phone, Q1 ideal avatar, Q2 niche, Q3 primary goal. Hidden: contact_id, anthology_id, stage. The producer and producer_email ride the webhook payload per the registry binding (the producer is the box owner; no separate producer identity system exists). Behavior: verify the route secret; validate hidden fields; tenant-check the location against the registry; dedup no-op on replay; upsert the participant on the composite key contact_id plus anthology_id (same person, second anthology equals second record, one contact_id); provision the Drive path Root, Producer, Anthology, Participant via drive-tree-provision.py under the EXISTING shared root (never a new root); CARD the participant onto the Anthology board (mc_board.py, fail-soft); acknowledge under 2 seconds; spawn the stage job detached. Unroutable submissions (typed reasons: unroutable_missing_ids, unknown_anthology, stage_mismatch, tenant_mismatch) land in the exceptions queue with the raw payload; never guessed, never dropped. Per-stage participant forms (tone, title choice, outline approval fallback, rewrite notes fallback) route the same way with stage validation against the ledger cursor. Replaces the LEGACY webhook, the getAll-limit-1 opportunity race (contact_id keying everywhere), and the 16-node folder subtree.

### S8 PACKAGE AND DELIVER (runs per deliverable as each stage completes, full sweep at participant completion; deterministic)
Per deliverable: (1) Google Doc via drive_adapter.py calling the Drive and Docs APIs DIRECTLY with the operator's EXISTING service account (the clawd/google-api.js pattern: service-account JWT, domain-wide impersonation under the GOOGLE_IMPERSONATE_USER label, full Drive scope) into the tree under the EXISTING anyone-can-read shared root; per-document sharing anyone-with-link VIEW only; (2) designed PDF via pdf_render.py (WeasyPrint-class) from the house template for that deliverable type, then guard-font-floor.py over the RENDERED file: no glyph below 14 point (the five legacy LLM HTML-formatter calls are RETIRED; their typography rules are harvested into the templates as CSS and markup, never model behavior); (3) upload Doc export and PDF to Convert and Flow media storage; (4) push both hosted links to the exact PRD Section 6 field keys via caf_delivery.py, keyed by contact_id, read back byte-for-byte; (5) update the three control fields (anthology_active_id, anthology_stage, anthology_rewrite_count); (6) the per-gate pipeline-stage update (Section 4); (7) completion notice through the sanctioned template only; (8) at participant completion, the signed process certificate (Skill 54 P7 pattern) and the board card to REVIEW via mc_board.py, promoted to done ONLY by the independent QC scorer at or above 8.5 (the engine never self-promotes).

### S9 ANTHOLOGY ASSEMBLY (new; TWO producer decisions bracket it: the ready-to-assemble trigger opens it, final sign-off closes it)
THE PRODUCER "I'M READY TO ASSEMBLE" TRIGGER (PRD 3.11, never automatic): when every participant is approved or EXPLICITLY excluded (an exclude approvals row), anthology_state.py ARMS the trigger (assembly_state not_ready to armed) and the approvals steward fires ONE readiness nudge to the producer. The producer fires the trigger from either door: the ASSEMBLY CARD on the Anthology board, or the readiness-nudge deep link; both doors hit the SAME endpoint and shell anthology_state.py record-approval --gate s9_ready --confirm-name. GUARDS enforced by the writer's legal-transition matrix, never by UI alone: (a) own-producer auth only (the box owner's Command Center session or the producer-scoped token); (b) every participant approved or explicitly excluded; (c) at least min_chapters FROZEN approved chapters (floor 2, per-anthology configurable up); (d) typed anthology-name confirmation (mismatch exits 5, nothing changes); (e) one-way: re-firing is an acknowledged no-op; reopening collection afterward is a producer-initiated exception that resets assembly_state and voids in-progress assembly. On success the anthology moves ready_to_assemble (assembly_state ready_confirmed) and the assembly job spawns.

Assembly steps: (1) ORDER CURATION (pin ae-01) applying the verified anthology craft: strongest pieces open and close, long-short alternation so long chapters never cluster, deliberate tone-shift management, subtheme grouping and paired contrasts; the proposal writes to the anthology record with rationale and the producer adjusts it on the Assembly card (assembly_state proposed, then adjusted); (2) editor's introduction and framing device in the producer's voice (pin ae-02), built ONLY from producer-supplied inputs, never fabricated; (3) front and back matter (pin ae-04); (4) contributor bios from ledger identities (pin ae-03); (5) compile the manuscript from FROZEN approved chapter artifacts, byte-identical by sha256, on the LONGCTX tier when configured, else chunked on HEAVY-WRITER (assembly_state compiled); (6) assembly-scope Gate B (every approved chapter present exactly once, order matches curation, introduction references only real contributors, one continuous 14-point-floor PDF); (7) producer FINAL SIGN-OFF on the Assembly card (gate s9_producer, a separate decision from the trigger; assembly_state signed_off); (8) manuscript Doc plus PDF delivered, manuscript fields pushed, participants move approved to delivered. This stage is what makes the product an anthology.

Cross-cutting at every stage: every transition through anthology_state.py (the board card and any rollup read the same ledger, never recompute into a second store); every billable call metered before and after; insufficient credits HOLDS durably with ONE deduped founder alert; duplicate deliveries acknowledge without a second run; no stage ever blocks on a human.

## 6. Failure handling (typed, every class)

- INSUFFICIENT CREDITS on any provider: hold_queue.py HOLDS the job with full state; one deduped founder alert through the gateway; the daily tick retries the chain; resume from the exact stage cursor. (The exact class that killed LEGACY twice.)
- MODEL CHAIN EXHAUSTED: hold plus alert, as above.
- GATE TIMEOUT: none exists; gates wait indefinitely; the board surfaces stuck-over-7-days highlighting, one deduped auto re-nudge fires at 7 days, and further re-sends are manual (rate-limited).
- CALLBACK LOST (Kie.ai): bounded re-poll, then hold plus alert.
- CONTENT GATE STRIKE-OUT (3 internal QC attempts): hold plus alert carrying the failing checks and best draft; standards never relaxed.
- REWRITE BUDGET EXHAUSTED: Approve as-is or producer escalation; a third rewrite is an illegal transition.
- UNROUTABLE SUBMISSION: exceptions queue, operator reconciliation, replay through S0.
- BOARD UNREACHABLE: fail-soft; the pipeline never blocks; cards reconcile on the daily tick.
- FOREIGN, EXPIRED, OR REPLAYED PARTICIPANT TOKEN: refused by gate_engine.py verify; nothing changes.
- CRASH ANYWHERE: the ledger is the truth; replaying the last event is a no-op or a clean resume.

## 7. Forms contract

All forms are Convert and Flow forms (universal intake plus per-stage participant forms), each carrying the three hidden fields contact_id, anthology_id, stage. The engine never hosts a form and never uses mail-embedded wait forms or approve-by-reply. Form identifiers bind per anthology in the anthology registry at provisioning. Field labels are harvested from LEGACY (tone influences times four, chapter_about, personal_stories, title and subtitle choice, change-request notes) so participants see continuity; Make.com and n8n variable syntax in the pinned prompt bodies maps deterministically to the canonical field dictionary (SPEC 6.2 and Appendix B), and the composer refuses any unresolved slot.

## 8. Provisioning and revocation

provision-anthology-client.sh (idempotent; config writes as the node user; every credential check reports SET or NOT SET, never a value): verify credentials by label (caf_credential_gate.py, all three env stores, live process env first, anti-commingling fingerprint); create or verify the PRD Section 6 custom fields by exact key (missing fields stop SETUP with an operator surface); AUTO-PROVISION the standard Anthology pipeline and stages in the client's own Convert and Flow account with the CLIENT'S OWN private integration token and bind the caf_stage_map into the registry (Section 4); register forms with hidden fields; SEED THE ANTHOLOGY DEPARTMENT in the client's Command Center (Skill 32 add-department.sh / POST /api/departments create:true, idempotent, read back and verified; the never-seeded-books-department defect is not repeated); provision the Drive producer folder under the operator's EXISTING shared delivery root (direct Drive API, existing service account, never a new root); bootstrap the ledger base and mirror; generate the webhook route and its secret; register the ONE daily tick in the cron inventory; run verify-webhook-t1-t9.sh; fire one smoke test (balance endpoints only, at or under one cent).

revoke-anthology-client.sh: invalidate every outstanding participant gate token, ARCHIVE the Anthology board cards, revoke Drive shares (view links regenerated), disable the webhook route, export the client's data (the ratified export path), archive ledger rows, verify by probing a revoked token link and the disabled route, and leave ZERO recurring jobs (guard-cron-inventory.py proves it).
